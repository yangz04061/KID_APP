# 幼儿英语启蒙 App 技术方案建议（v1）

## 1. 文档目标

本文档给出首期网页端产品的技术方向、项目结构建议和实现边界，目标是在保证开发效率的前提下，优先完成适合 iPad 的稳定 MVP。

## 2. 技术选型结论

首期建议采用前后端分离中的“纯前端静态应用”方案。

推荐方案：

- 前端框架：React。
- 构建工具：Vite。
- 开发语言：TypeScript。
- 样式方案：CSS Modules 或普通 CSS 文件。
- 状态管理：React Context + 本地组件状态。
- 路由：React Router。
- 本地存储：localStorage。
- 部署方式：静态站点部署。

选择原因：

- 首期无登录、无后台、无复杂数据流，没必要引入服务端。
- Vite + React 启动快、开发效率高。
- TypeScript 便于后续扩展内容配置和游戏状态。
- 本地存储足以支撑 MVP 进度保存。

## 3. 不建议的首期方案

- 不建议一开始做原生 iPad App。
- 不建议引入复杂后端服务。
- 不建议上来就做多账户体系。
- 不建议引入重量级状态管理库。

原因：

- 会增加开发和维护成本。
- 不利于快速验证儿童交互是否有效。
- 首期主要风险在内容和体验，而不是系统复杂度。

## 6. 核心数据流建议

### 6.1 启动流程

- 应用启动。
- 读取本地设置和本地进度。
- 加载主题和单词配置。
- 渲染首页。

### 6.2 学习流程

- 用户选择主题。
- 根据主题读取单词预览内容。
- 顺序进入 3 类小游戏。
- 每完成一关更新当前进度。
- 完成主题后写入本地存储。

### 6.3 家长设置流程

- 家长进入家长区。
- 修改背景音乐、自动播放、中文释义等设置。
- 设置即时写入本地存储。

## 7. 关键实现建议

### 7.1 音频播放

- 每个单词发音都需要可重复播放。
- 首次用户交互后再正式触发连续音频，兼容浏览器自动播放限制。
- 尽量封装统一 audio manager，避免页面各自处理音频状态。

### 7.2 场景点击

- 找一找类游戏建议优先用绝对定位热点区域实现。
- 首期不要上复杂 canvas 交互。
- 先用简单、可维护的点击热区验证体验。

### 7.3 进度存储

- 首期统一使用 localStorage。
- 需要定义版本号字段，便于后续升级数据结构。
- 加载失败时要有默认初始值，避免白屏。

### 7.4 家长入口

- 首期可以用长按 2 秒进入。
- 或通过角落小按钮 + 二次确认进入。
- 不建议复杂验证码，避免破坏使用节奏。

## 8. 响应式与 iPad 适配建议

- 以 iPad 横屏为第一设计基准。
- 保证 1024px 左右宽度下布局舒适。
- 对手机竖屏做简化适配。
- 点击热区至少 44px 以上。
- 避免依赖 hover 行为。

## 9. 性能与稳定性要求

- 首屏资源控制，避免一次加载全部大图。
- 主题资源按需加载。
- 音频文件压缩到适合网页播放的大小。
- 动画使用 CSS 或轻量方案，避免过重库。
- 页面切换和音频播放异常时要有兜底逻辑。

## 10. 开发阶段建议

### 第一阶段：骨架搭建

- 创建 React + Vite + TypeScript 项目。
- 配置路由、基础样式、内容 JSON。
- 完成欢迎页、首页、主题页骨架。

### 第二阶段：核心流程实现

- 实现单词预览页。
- 实现 3 类小游戏。
- 实现主题完成页。

### 第三阶段：家长区与本地存储

- 实现学习进度记录。
- 实现设置读写。
- 实现家长入口。

### 第四阶段：内容接入与打磨

- 接入正式图片和音频素材。
- 调整交互反馈。
- 做 iPad 实机适配。

## 11. MVP 开发优先级

### P0

- 首页到主题页的主路径。
- 单词预览。
- 听声音选图片。
- 本地进度保存。

### P1

- 图片配对。
- 找一找 / 点一点。
- 家长区设置。
- 奖励反馈。

### P2

- 更丰富动画。
- 更多主题扩展能力。
- PWA 封装。

## 12. 后续可扩展方向

- 增加更多主题包。
- 增加跟读录音。
- 增加离线缓存。
- 增加多设备同步。
- 增加内容后台配置能力。

## 13. 下一步建议

1. 如果直接进入开发，建议先从 React + Vite + TypeScript 脚手架开始。
2. 先用占位图和占位音频跑通完整流程，再替换正式素材。
3. 开发时优先实现一条主题完整闭环，再复制到其他主题。
